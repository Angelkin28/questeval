# Cambios del 11 de Marzo de 2026

## Resumen de sesión

Esta sesión se enfocó en tres áreas: integridad de datos al eliminar usuarios, seguridad del backend (eliminar endpoint peligroso) y el flujo completo de rúbricas para maestros.

---

## 1. Integridad de Datos — Eliminación en Cascada de Proyectos

**Archivo:** `Backend/Services/UsersService.cs`

**Problema:** Al eliminar un usuario, sus proyectos quedaban como "basura" en la base de datos sin dueño ni integrantes (registros huérfanos).

**Solución implementada:**

Cuando se elimina un usuario, ahora se ejecuta la siguiente lógica sobre proyectos:

### Proyectos donde el usuario es dueño (`UserId`)
- Si el proyecto **no tiene `TeamMembers`** → se elimina el proyecto completo.
- Si el proyecto **aún tiene integrantes** → solo se limpia el campo `UserId` del proyecto (el proyecto sobrevive con sus integrantes).

### Proyectos donde el usuario aparece como integrante (`TeamMembers[]`)
- Se remueve el nombre del usuario del array `TeamMembers`.
- Si tras la remoción el proyecto queda **sin dueño y sin integrantes** → se elimina automáticamente.
- Si aún hay otros integrantes o dueño → el proyecto sobrevive actualizado.

**Regla de negocio establecida:** Un proyecto solo puede existir si tiene al menos un miembro (dueño o integrante de equipo).

## 2. Seguridad — Eliminación de Endpoint Peligroso

**Archivos modificados:**
- `Backend/Controllers/AdminController.cs`
- `Backend/Services/Interfaces/IProjectsService.cs`
- `Backend/Services/ProjectsService.cs`

**Descripción:** Se propuso inicialmente un endpoint `DELETE /api/Admin/projects/all` para limpiar la base de datos. Este endpoint fue **eliminado por ser demasiado peligroso** (operación irreversible sin confirmación, expuesta via HTTP).

Se eliminaron también:
- El método `DeleteAllAsync()` de `IProjectsService` e implementación en `ProjectsService`
- La inyección de `IProjectsService` en `AdminController` que solo servía para ese propósito

## 3. Sistema de Rúbricas — Flujo Completo para Maestros

### 3a. Corrección crítica del evaluador

**Archivo:** `Frontend/src/app/rubric/page.tsx`

**Problema:** Al guardar una evaluación, se enviaba `evaluatorId: "current_user"` como placeholder. El backend buscaba este ID en la base de datos, fallaba y no guardaba la evaluación.

**Solución:** Ahora se lee el `userId`/`id` real del usuario desde `localStorage`:

```ts
const currentUser = JSON.parse(localStorage.getItem('user')!);
const evaluatorId = currentUser.userId ?? currentUser.id;
```

Si no hay sesión activa, redirige a `/login` en lugar de fallar silenciosamente.

### 3b. Guard de rol en editor de rúbrica

**Archivo:** `Frontend/src/app/rubric/edit/page.tsx`

**Problema:** Cualquier usuario podía acceder a `/rubric/edit` y crear/editar/eliminar criterios de evaluación globales.

**Solución:** Se añadió un `useEffect` de verificación al montar el componente:
- Si no hay sesión → redirige a `/login`
- Si el rol no es `Profesor` ni `Admin` → redirige a `/dashboard`

### 3c. Acceso directo desde el dashboard del maestro

**Archivo:** `Frontend/src/app/dashboard/components/TeacherDashboard.tsx`

Se añadió un botón **"Rúbrica"** junto al botón "Nuevo Grupo" en el header del dashboard del maestro. Navega a `/rubric/edit` para gestionar los criterios de evaluación.

## Flujo completo de rúbricas (documentado)

```
1. Maestro define criterios
   → Dashboard → botón "Rúbrica" → /rubric/edit
   → CRUD: nombre, descripción, puntaje máximo por criterio
   → POST/PUT/DELETE /api/Criteria  (solo Profesor/Admin)

2. Alumno entrega proyecto
   → El alumno cambia status del proyecto a 'Completed'

3. Proyecto aparece en "Por Calificar"
   → TeacherDashboard muestra proyectos con status 'Completed' en sus grupos

4. Maestro califica
   → Click en proyecto → /rubric?projectId={id}
   → Slider por cada criterio (0 a MaxScore)
   → FinalScore = promedio de todos los scores
   → Guarda en POST /api/Evaluations con userId real del maestro

5. Alumno ve su calificación
   → Dashboard → sección "Mis Evaluaciones"
   → Muestra score por proyecto con código de color (verde ≥80, amarillo ≥60, rojo <60)
```
