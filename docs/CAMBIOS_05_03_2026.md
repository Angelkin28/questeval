## 2. Contadores incrementales de usuarios

### Comportamiento definido
- Los contadores **nunca retroceden**: si se crea el usuario 2, se elimina y se crea otro, ese nuevo recibe la ID **3**, no vuelve a ser 2.
- El contador usa `$inc` en MongoDB (solo suma hacia adelante).
- Se eliminó el botón "Reiniciar IDs" del panel de admin — el reseteo puntual se hace vía `POST /api/Admin/reset-counters` desde Postman/Swagger con token de admin.

### Limpieza de base de datos realizada
- Se eliminaron todos los usuarios excepto el admin (`won.dorado.mid@gmail.com`).
- Se estableció el `UserId` del admin en **`"1"`**.
- Se reseteó el contador de `users` a **`1`** → el próximo usuario registrado recibe la ID **2**.

## 3. Validación de correos institucionales

### Dominios permitidos
| Rol | Dominio requerido |
|---|---|
| Alumno | `@alumno.utmetropolitana.edu.mx` |
| Maestro/Profesor | `@utmetropolitana.edu.mx` |

### Implementación
- La validación usa `split('@')[1]` para comparación exacta del dominio, evitando cualquier bypass con subdominios.
- Los mensajes de error al usuario son:
  - Alumno: **"Los alumnos deben usar su correo Institucional"**
  - Maestro: **"Los Maestros deben usar su correo Institucional"**
- El placeholder del campo de correo muestra: **"Correo institucional"**

### Excepción: cuentas de prueba (test)
- Correos que terminen exactamente en `testquesteval@gmail.com` (ej. `alumno1testquesteval@gmail.com`, `maestro1testquesteval@gmail.com`) **se saltan la validación de dominio**.
- Estas cuentas se crean automáticamente con `EmailVerified: true` y `VerificationStatus: "approved"` en el backend.
- Al registrarse, van directo a `/login` sin pasar por la pantalla de verificación OTP.
- Correos genéricos de Gmail como `angel.rosado@gmail.com` **siguen siendo rechazados** — solo se acepta la terminación exacta `testquesteval@gmail.com`.

## 5. Corrección de endpoint `GET /api/Users/{id}`

**Problema:** Al buscar por ID incremental (ej. `/api/Users/3`), el driver de MongoDB intentaba parsear `"3"` como ObjectId → lanzaba `FormatException` → 500 Internal Server Error.

**Solución en `UsersService.GetByIdAsync`:**
```csharp
if (!MongoDB.Bson.ObjectId.TryParse(id, out _))
    return null;
```
- Si el valor no es un ObjectId válido, retorna `null` sin explotar.

**Solución en `UsersController.GetById`:**
```csharp
var user = await _service.GetByIdAsync(id)
        ?? await _service.GetByUserIdAsync(id);
```
- Ahora acepta tanto el MongoDB ObjectId como el UserId incremental (ej. `"3"`).

**Campos faltantes corregidos:** El endpoint también ahora devuelve `UserId`, `EmailVerified` y `VerificationStatus` que antes faltaban en la respuesta.

## 7. Corrección de evaluaciones como invitado

**Problema:** El frontend enviaba los campos con nombres incorrectos al backend.

| Frontend enviaba | Backend esperaba |
|---|---|
| `evaluatorId` | `userId` |
| `criterionId` (en detalles) | `criteriaId` (en detalles) |

**Solución:** El método `api.evaluations.create()` en `api.ts` ahora mapea los campos correctamente antes de enviar al backend:
```typescript
const body = {
    projectId: data.projectId,
    userId: data.evaluatorId,
    details: data.details.map(d => ({
        criteriaId: d.criterionId,
        criterionName: d.criterionName,
        score: d.score,
    })),
};
```
